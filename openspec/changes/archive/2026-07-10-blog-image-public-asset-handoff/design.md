## Context

Flow A blog publish calls `ensure_blog_image()` before `validate_ready_post()` when canonical image prerequisites are missing and ComfyUI generation is enabled (`comfyui-blog-image-generation`). Today that step:

- Detects missing canonical prerequisites using editorial ready Markdown + sibling PNG only.
- May call ComfyUI and write `blog-posts/ready/<source_slug>.png`.
- Updates front matter to `image: /assets/images/<public_slug>.png`.
- Explicitly does **not** write to the public GitHub Pages checkout.

The GitHub Pages bridge (`github_pages_publish.py`) copies the ready PNG into `assets/images/<public_slug>.png` during `run_publish(apply=True)`, but only after validation and lifecycle transitions. Production failure on post `02-deferring-is-not-avoiding-it-can-be-architecture` showed ComfyUI had already written a valid editorial PNG while the public asset remained missing and publish aborted with generic `blog_image_generation_failed`. Retries risk duplicate ComfyUI calls and operators were tempted to manually copy images into `silverberdi.github.io`.

Constraints:

- Worker HTTP boundary only; no n8n Execute Command (ADR-0001).
- Public repo path from `SILVERMAN_GITHUB_PAGES_REPO_PATH`.
- No `git commit` / `git push` in the public repo.
- Preserve existing Flow A scope: blog publish only, no LinkedIn, no calendar/n8n activation.
- Tests use fakes; no live ComfyUI; no root-owned file assumptions.

## Goals / Non-Goals

**Goals:**

- Unify editorial image readiness with public asset readiness before validation/publish.
- Treat existing readable public `assets/images/<public_slug>.png` as authoritative for Jekyll; satisfy public publishing prerequisites without ComfyUI when it exists.
- Never overwrite an existing public asset during handoff or reuse.
- Backfill missing ready sibling PNG from public asset only for bridge compatibility; never overwrite an existing ready sibling PNG.
- Prefer non-blocking warnings when public asset exists but ready sibling backfill fails, if publish can still proceed safely using the public asset.
- Adopt sibling `blog-posts/ready/<source_slug>.png` into the public repo when editorial image exists but public asset does not (post `02` bug fix).
- Hand off ComfyUI-generated bytes to the public repo in the same orchestration step.
- Fail with `blog_image_public_asset_handoff_failed` only when required copy/adoption **into public assets** fails (not when optional ready-sibling backfill fails while public asset is sufficient).
- Skip ComfyUI on retry when a reusable local sibling PNG or public asset already exists.
- Write copied public assets with permissions readable by the blog repo checkout user/process.
- Record handoff source (`existing_public_asset`, `ready_sibling_png`, `comfyui_generated`) and optional backfill warnings in `blog_image_generation` metadata.

**Non-Goals:**

- LinkedIn publication, n8n activation, cron, calendar changes.
- Modifying archived OpenSpec changes or `linkedin-article-preview-image-support`.
- Overwriting mismatched public images during reconciliation (existing `public_image_adopted` behavior in publish reconciliation remains separate).
- Changing bridge non-overwrite rules for `_posts/` during normal first-time publish.

## Decisions

### 1. Extend `ensure_blog_image()` rather than a separate publish-only step

**Decision:** Add public asset evaluation and handoff inside `blog_image_generation.py` (same entry point `publish_blog_post` already calls), accepting `github_pages_repo_path` (or resolving from env).

**Rationale:** Image prerequisites are incomplete until both editorial ready PNG **and** public `assets/images/<public_slug>.png` exist (or are provably reusable). Keeping orchestration in one place preserves pre-validation ordering and avoids duplicate detection logic in `blog_publish_flow.py`.

**Alternative considered:** Handoff only inside `github_pages_publish.run_publish` apply. Rejected because publish already failed before bridge apply in the production scenario; prerequisites must be satisfied earlier.

### 2. Detection order: public asset → ready sibling → ComfyUI

**Decision:** When evaluating image prerequisites:

1. If public repo `assets/images/<public_slug>.png` exists and is a readable regular file → public/Jekyll prerequisite is satisfied; ensure front matter is canonical; do **not** call ComfyUI; do **not** overwrite the public asset. Optionally backfill `blog-posts/ready/<source_slug>.png` from the public asset **only when the ready sibling is missing** (never overwrite an existing ready sibling). If backfill fails, record a non-blocking warning and continue when publish can proceed safely using the public asset; fail only if downstream validation/publish truly requires the ready sibling.
2. Else if ready sibling PNG exists → adopt: copy to public `assets/images/<public_slug>.png` when missing, patch front matter if needed; do **not** call ComfyUI.
3. Else if generation enabled → ComfyUI generate editorial PNG, then hand off to public assets.
4. If required public-asset handoff copy fails at steps 2 or 3 → `blog_image_public_asset_handoff_failed`. Ready-sibling backfill failures at step 1 are warnings unless validation cannot proceed.

**Rationale:** Public asset is canonical for Jekyll; missing ready sibling alone must not trigger ComfyUI or public overwrite. Backfill is compatibility-only. Post `02` adoption path remains the primary production fix.

### 3. Thin public asset copy helper in `github_pages_publish.py`

**Decision:** Add an internal helper (for example `copy_public_blog_image(source_png, public_repo_path, public_slug)`) reused by image orchestration. Do not duplicate slug/path rules; derive targets via existing `IMAGES_RELATIVE` and `resolve_public_slug`.

**Rationale:** Single source of truth for `assets/images/<public_slug>.png` paths aligned with bridge conventions.

**Alternative considered:** Inline `shutil.copy2` in `blog_image_generation.py`. Rejected to avoid drift from bridge path validation.

### 4. Permission-safe copy semantics

**Decision:** When creating or replacing the public asset during handoff (only when target missing), write via atomic temp file + rename in target directory; set file mode `0o644` (umask-respecting). Do not require post-copy `chown` for normal success. Tests create files as the test user.

**Rationale:** Docker worker and host checkout may run as non-root; readable assets suffice for Jekyll/git operations by the repo owner.

**Trade-off:** If parent `assets/images/` is not writable by the worker user, handoff fails with `blog_image_public_asset_handoff_failed` — operator fixes directory permissions once rather than per-post manual copy.

### 5. Do not overwrite existing public or ready sibling assets

**Decision:** If `assets/images/<public_slug>.png` already exists, reuse it (decision 2 step 1). Handoff copy into public assets runs only when the public target is missing. Ready sibling backfill from public asset runs only when `blog-posts/ready/<source_slug>.png` is missing; never overwrite an existing ready sibling PNG.

**Rationale:** Aligns with `github-pages-blog-publishing` non-overwrite policy and existing publish reconciliation that adopts differing public images rather than overwriting. Ready sibling backfill is optional compatibility, not authoritative over existing editorial files.

### 6. Distinct error code for handoff failures

**Decision:** Introduce `blog_image_public_asset_handoff_failed` in image orchestration; map to publish `errors[]` and `blog_image_generation.error_code`. Reserve `blog_image_generation_failed` for unexpected/generic failures only.

**Rationale:** Operators and execution connector can distinguish permission/path failures from ComfyUI outages.

### 7. Execution connector unchanged unless validation fails

**Decision:** No `editorial-calendar-flow-a-execution-connector` code changes; improved error code flows through existing `publish_blog_post` result JSON (`errors[]`, `blog_image_generation`).

**Rationale:** User scope limits connector changes to strict necessity.

### 8. Metadata extensions

**Decision:** Extend `blog_image_generation` summary with:

- `public_asset_handoff_status`: `reused` | `copied` | `not_needed` | `failed`
- `public_asset_source`: `existing_public_asset` | `ready_sibling_png` | `comfyui_generated`
- `public_repo_image_relative_path`: `assets/images/<public_slug>.png`
- `ready_sibling_backfill_status` (when applicable): `copied` | `not_needed` | `failed`
- `warnings[]` (optional): include `ready_sibling_backfill_failed` when backfill fails but publish proceeds using the public asset

**Rationale:** Supports ops debugging and tests without storing image bytes in metadata; distinguishes required public handoff failures from optional backfill warnings.

## Risks / Trade-offs

- **[Risk] Worker lacks write permission on public repo checkout** → Mitigation: clear `blog_image_public_asset_handoff_failed`; document expected directory ownership in ops doc; validate repo layout before copy.
- **[Risk] Public and ready PNGs differ in content** → Mitigation: when public asset exists, reuse public as canonical for Jekyll; optionally backfill ready sibling from public for bridge consistency only when missing; do not overwrite public asset or existing ready sibling.
- **[Risk] Backfill failure blocks publish unnecessarily** → Mitigation: treat backfill as non-blocking when public asset satisfies Jekyll prerequisites; surface warning in metadata; fail only when validation truly requires ready sibling.
- **[Risk] Duplicate logic with bridge apply image copy** → Mitigation: shared helper; bridge apply may no-op image copy when handoff already placed file (bridge non-overwrite still safe).
- **[Risk] ComfyUI called on retry despite local PNG** → Mitigation: explicit detection order; tests for retry after partial failure.

## Migration Plan

1. Implement helper + extended `ensure_blog_image()` behind this change.
2. Deploy worker container with unchanged env (`SILVERMAN_GITHUB_PAGES_REPO_PATH` must be writable).
3. Retry failed calendar item `2026-07-10-deferring-is-not-avoiding-it-can-be-architecture` — expect adoption of existing ready PNG and successful handoff without ComfyUI.
4. Rollback: revert worker image; no data migration required (public assets already written remain).
